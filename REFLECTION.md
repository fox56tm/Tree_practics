
## 1. Публичный интерфейс
В публичный интерфейс множества должны входить только те функции, которые описывают поведение множества как математического объекта,например:
- `void bstInsert(BST* tree, int value)`
- `bool bstContains(BST* tree, int value)`
- `void bstInorder(BST* tree)` (возможно стоит оставить 1 из обходов) 
- `void bstPreorder(BST* tree)`
- `void bstPostorder(BST* tree)`
- `int bstSize(BST* tree)`
- `int bstMin(BST* tree)` 
- `int bstMax(BST* tree)` 
- `void bstDelete(BST* tree, int value)`
- `BST* bstMerge(BST* tree1, BST* tree2)`
## Детали реализации
В деталях должны оставаться то, что касается именно структуры BST, а также вспомогательные функции:
- `static bool isValidHelper(Node* node, int min, int max)` вспомогательная для проверки валидности дерева
- `static int inorderKth(Node* node, int k)` — внутренний обход для поиска k-го элемента
- `static void push(StackNode** top, Node* node)` внутренняя структура для стека итератора
- `static Node* pop(StackNode** top)` внутренняя функция извлечения из стека
- `static void pushLeft(Iterator* it, Node* node)` вспомогательная для инициализации итератора
## 2. Функции, не имеющие смысла в интерфейсе множества
- `bstKthMin` — поиск k-го минимума не имеет смысла для абстрактного множества. В множестве важен факт наличия элемента, а не его порядковый номер.
- Функции, возвращающие указатели на `Node`, так как это раскрывает внутреннее устройство.
- `bstKthMin` Зависит от упорядоченности BST. В хеш-таблице понятие "k-й минимальный" отсутствует.
- `iteratorInit`, `iteratorNext`, `iteratorHasNext`, `iteratorFree` — Итератор — отдельный механизм обхода, не входит в интерфейс самого множества. Должен быть вынесен в отдельный модуль. 
`bstIsValid` Диагностическая функция для проверки корректности BST. В готовом продукте не нужна.

## 3. Замена на другое
- В заголовочном файле можно использовать `typedef struct Set Set;`, а саму структуру описывать только в `.c` файле (Opaque pointer).
- Использовать общие типы данных, не привязанные к деревьям.
